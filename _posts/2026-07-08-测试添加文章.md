### 核心数据流（一条消息的完整生命周期）

拿你刚发的那句"openclaw是什么"举例：

```
Step 1: 通道接收
─────────────────
WebChat → WebSocket → Gateway HTTP Server
                                        ↓
Step 2: 会话解析  
─────────────────
SessionManager 找到或创建 session
分辨是哪个用户、哪个通道、之前聊过什么
                                        ↓
Step 3: 入队
─────────────────
放入 session 队列（per-session lane）
确保串行处理，不会冲突
                                        ↓
Step 4: 上下文组装（Context Engine）
─────────────────
读取：
  - 系统提示词模板（dist/gateway 内置）
  - SOUL.md → "我是谁、什么性格"
  - AGENTS.md → "操作规则"
  - MEMORY.md → "之前记住的事"
  - Skills 的 SKILL.md → "可用工具说明"
  - 会话历史 → "刚才聊了什么"
  - 当前时间、时区、运行时信息

拼成一个完整的 prompt 包
                                        ↓
Step 5: 调大模型
─────────────────
通过 Provider Extension（比如 anthropic 插件）
发送 prompt + 工具定义给 API
                                        ↓
Step 6: 模型推理
─────────────────
模型理解了你的问题 → 决定生成文字回答
（不需要调工具，所以没 Tool Call）
                                        ↓
Step 7: 流式输出
─────────────────
模型输出文本 → 通过 WebSocket stream 逐段推送到 WebChat
你看到的就是实时流式显示
                                        ↓
Step 8: 持久化
─────────────────
这条消息 + 模型回复写入会话文件
写入 memory/YYYY-MM-DD.md（如果模型决定要记）
                                        ↓
Step 9: 后处理
─────────────────
插件 hook 触发（agent_end）
Context Engine 收尾
```

### 关键实现细节

**1. Gateway 协议（WebSocket）**

不是 HTTP REST API，而是长连接 WebSocket。为什么？

- 双向推送：模型在跑的时候，Gateway 可以逐字推送"正在思考"的状态
- 实时性好：不像 HTTP 那样每次请求都要重新建立连接
- 适合 Agent 场景：模型可能会调多个工具，每个工具调用都需要实时返回状态

协议帧长这样：
```json
// 客户端发给 Gateway
{"type":"req", "id":"req_123", "method":"agent", "params":{"message":"你好"}}

// Gateway 推送给客户端
{"type":"event", "event":"agent", "payload":{"stream":"assistant", "delta":"正在..."}}
{"type":"event", "event":"agent", "payload":{"stream":"tool", "tool":"read_file", "status":"start"}}
{"type":"res", "id":"req_123", "ok":true, "payload":{"status":"ok"}}
```

**2. Agent Loop（pi-embedded-runner）**

这是最核心的部分。不是一个 for 循环那么简单，而是：

- **Context Engine 插件化**：默认用 `legacy` 引擎（就是简单的截断+摘要），也可以换成第三方引擎来控制上下文怎么组装、怎么压缩
- **Tool Execution**：模型返回 Tool Call → Gateway 截获 → 在本地执行 → 结果塞回给模型。这步是有**序列化队列**的，防止工具调用冲突
- **Timeout 控制**：可以设置单次推理超时、整个对话超时
- **生命周期事件**：每条消息都是 `start → think → tool_call → tool_result → think → ... → end` 这样一串事件

**3. 插件体系（Extensions）**

OpenClaw 的插件系统是它的"模块化心脏"。每个插件通过统一的 API 注册：

```typescript
// 插件可以注册的东西（伪代码）
api.registerProvider("anthropic", {...})    // 模型提供商
api.registerChannel("telegram", {...})      // 消息通道  
api.registerHook("before_tool_call", {...}) // 生命周期钩子
api.registerTool("hue_set_light", {...})    // 新工具
api.registerContextEngine("my-engine", {...}) // 上下文引擎
```

当前有 **40+ 模型插件**、**15+ 通道插件**、若干功能插件。每个都是独立的 npm 包级别的模块。

**4. Skills 不是插件——是提示词**

很多人搞混这个。Skills 和 Extensions 的区别：

- **Extension（插件）**：注册真正的工具实现 + 代码逻辑。比如天气插件会调用天气预报的 API。
- **Skill**：只是一个 `SKILL.md` 文件，教 AI **怎么用**已有的工具。比如 Skill 里写"查天气预报时先调这个 API，然后这样解析结果"。

Skills 的好处是不用写代码——写个 Markdown 文件就能扩展 AI 的能力。但 Skills 不能凭空造工具，工具必须由插件或 Gateway 本身提供。

**5. 会话 + 记忆**

- **Session**：每次对话的完整记录。存在 `~/.openclaw/sessions/` 下，按通道+用户分类
- **Memory**：长期记忆。`MEMORY.md` + `memory/*.md`，纯 Markdown 文件。AI 自己读、自己写
- **记忆搜索（memory_search）**：混合搜索（向量相似度 + 关键词匹配），需要 embedding API

### 回头看你说的话

> openclaw本质是一个网关，其上游是大模型，下游是具体的数据

从这个白盒角度看，更精确的表述应该是：

**OpenClaw 是一个事件驱动的 AI Agent 运行时 + 网关。**

- **运行时**：它有完整的 Agent Loop（接收→理解→决策→行动→反馈循环），不只是转发
- **网关**：它在多个消息通道之间做路由和协议转换
- **插件化**：上游可以换模型（40+ 选择），下游可以接任意工具（通过插件+exec+API），中间的行为可以被插件 hook 拦截修改

这不只是一个"转发器"——它是在"托管一个 AI Agent 的完整生命周期"。
